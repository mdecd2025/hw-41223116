    Important Vocabulary and Phrases (重要詞彙與短語)

    1. Simulation - 模擬：一個模仿現實世界過程或系統的過程。
    2. Prototyping - 原型製作：創建產品或系統的早期版本（原型）的過程。
    3. Mobile Robot - 移動機器人：能夠在其環境中移動的機器人，通常配有感測器和驅動器。
    4. Locomotion Schemes - 移動方式：機器人移動的方式或系統（例如，輪式機器人、步態機器人、飛行機器人等）。
    5. Sensors - 感測器：用來檢測並對物理刺激作出反應的裝置，例如相機或距離感測器。
    6. Actuators - 驅動器：執行動作的裝置，如馬達或車輪。
    7. Controller Program - 控制程式：控制機器人行為或運作的程式。
    8. Interface - 介面：系統之間的連接或溝通（在此指模擬機器人與實體機器人之間的介面）。
    9. Multi-Agent Research - 多代理研究：涉及多個機器人共同工作的研究，通常是協作或集群設定。
    10. Adaptive Behavior - 自適應行為：基於環境或經驗變化的行為（通常涉及如神經網絡等人工智慧方法）。
    11. BotStudio - BotStudio：一種簡單的圖形化程式設計語言，用於在Webots中編程機器人。
    12. VRML97 (Virtual Reality Modeling Language) - VRML97（虛擬現實建模語言）：描述3D物體和環境的標準。
    13. PROTOTYPE Files - PROTOTYPE檔案：定義模擬中新增物件的檔案。
    14. World File - 世界檔案：定義模擬中環境、物體和機器人的檔案。
    15. Controller Directory - 控制器目錄：儲存機器人控制程式源碼和二進位檔案的目錄。
    16. Supervisor Controller - 監督控制器：一種特殊的控制器，具有更高的許可權來管理模擬功能。
    17. Binary Executables - 二進位執行檔：已編譯並可在系統中運行的程式檔案（例如".exe"檔案）。
    18. Interpretation vs. Compilation - 解釋與編譯：指編程語言的處理方式（例如，編譯語言如C/C++與解釋語言如Python的區別）。

  1. Grammar Explanation (文法說明)

  1. Overall Grammar Features (整體文法特點)
  • Present Simple Tense (現在簡單時): 用來描述事實或一般性資訊（例如："Webots is..."）。
  • Modal Verbs (情態動詞)（can, may, would）: 表示可能性、能力或請求（例如："You can program..."）。
  • Relative Clauses (關係子句): 提供名詞的額外資訊（例如："robots called mobile robots"）。

HW1相關提問:

Webots 是什麼樣的軟體？它主要用途為何？

Webots 是一套由 Cyberbotics 所開發的 開源 3D 機器人模擬軟體，用於模擬機器人在真實世界中行為的虛擬環境。其主要用途包括：

  • 設計與模擬移動式或靜態機器人系統

  • 測試機器人控制演算法（例如感測器融合、路徑規劃、強化學習等）

  • 教學與研究（機器人學、人工智慧、自主導航等）

  • 開發與驗證機器人軟硬體設計（可搭配真實機器人）

如何在 Webots 中建立並執行一個新的模擬世界（world）？

  • 啟動 Webots。

  • 選單中點選 File → New World 建立新的模擬場景。

  • 可透過左側「場景樹」(Scene Tree) 加入環境物件與機器人，例如：

    • 加入地面：Add > Solid > Floor

    • 加入機器人：Add > Robot

  • 編輯物件屬性，如大小、材質、位置等。

  • 儲存世界檔案（副檔名為 .wbt）。

  • 點選「播放」按鈕（上方工具列 ▶️），即可執行模擬。

請簡述 Webots 控制器（controller）的功能及其與機器人的關係。

  • 控制器是 Webots 中操控機器人行為的程式碼（通常是 C、Python、C++ 等語言撰寫）。

  • 它直接與機器人的感測器與致動器互動（例如讀取紅外線感測器、控制馬達轉速）。

  • 每個控制器程式通常對應一個機器人，可在 robot 物件的 controller 欄位指定對應的控制器名稱。

  • 控制器執行週期性迴圈（如 while robot.step(timestep) != -1:）來模擬真實時間控制。

Webots 中的 "Supervisor" 是什麼？它與一般機器人控制器有何不同？

  • Supervisor 是一種特殊的控制器，擁有 更高層次的控制權限。

  • 除了能像一般控制器一樣控制機器人，它還能：

    • 修改場景中的物件（新增、刪除、移動）

    • 取得所有物件的位置與狀態

    • 重設模擬、儲存世界狀態、載入新的世界

    • 用於強化學習（如重置環境）

  • 適用於：全局管理、多機器人系統、評分系統、自動測試場景等應用

如何將 Webots 世界（world）或機器人設定檔分享給他人，確保對方可順利開啟模擬

  • 其他依賴檔案（貼圖、模型、資料檔）

    • worlds/：模擬場景檔（.wbt）

    • protos/：自訂機器人或物件的描述檔（.proto）

    • controllers/：控制器程式碼（每個控制器一個子資料夾）

    • 其他依賴檔案（貼圖、模型、資料檔）

    打包方式：

    1. 使用 .zip 或 .tar.gz 壓縮整個專案資料夾。

    2. 確保分享者與接收者都安裝有相同版本的 Webots。

    3. 接收者只需解壓縮後，透過 Webots 開啟 .wbt 檔即可。

Webots 支援哪些主流程控制語言？各有何優缺點？

語言 優點 缺點
C/C++ 高效能，資源控制佳，適合嵌入式模擬與真實硬體整合 開發速度慢、語法較繁瑣
Python 撰寫快速、易讀、資源豐富（支援 NumPy、PyTorch） 效能略低，不適合重度運算
Java 跨平台佳，結構清晰 程式碼稍長、社群支援較少
MATLAB 適合科學計算與控制理論模擬 授權成本高，與 Webots 整合需額外安裝套件

請說明如何使用 Webots 的 Console 與 Debug 工具協助程式錯誤排除。

  • Console（控制器輸出視窗）：

    • 顯示控制器 print() 或 printf() 的輸出。

    • 顯示錯誤訊息、例外（Exception）與除錯訊息。

    • 可使用 robot.getName() 等方法輸出除錯資訊。

  • Debug 工具（在 IDE 或外部開發工具中）：

    • 可將控制器連結至外部 IDE（如 Visual Studio Code）來設斷點除錯。

    • C/C++ 控制器可透過 gdb 除錯。

    • Python 控制器可加入 import pdb; pdb.set_trace() 進入互動除錯模式。


  • Supervisor 支援除錯：

    • 可檢查世界中的物件狀態與位置變化。

    • 可設定模擬暫停、單步執行等功能。



Tutorial3 相關提問:


1. 如何讓機器人偵測地面顏色？

在 Webots 中，使用地面感測器（Ground Sensors）來偵測地面顏色或反射率。這些感測器通常裝設在機器人底部，透過模擬的光學反射值來讀取不同顏色的地面資訊。


2. Ground Sensors 在 Webots 裡的作用為何？

Ground Sensors 模擬了現實中的 反射光感測器，其作用為：

  • 偵測地面表面顏色（如黑、白、灰）

  • 協助機器人辨認路線（如循線行走）

  • 用於場地區辨（例如偵測進入特定區域）

這些感測器會回傳浮點數值，代表地面反光程度。


3. 如何在程式中取得 Ground Sensor 的數值？

以 Python 為例：

python
ground_sensor = robot.getDevice('gs0') # 取得地面感測器裝置
ground_sensor.enable(timestep) # 啟用感測器，設定取樣週期

while robot.step(timestep) != -1:
value = ground_sensor.getValue() # 讀取感測器數值
print("Ground sensor value:", value)

注意：常見 e-puck 使用三個地面感測器：gs0, gs1, gs2，分別對應左、中、右。


4. Tutorial3 中的 e-puck 機器人如何根據地面顏色改變行為？

在 Webots 的 Tutorial 3 中：

  • e-puck 機器人使用 三個 ground sensors

  • 程式會判斷讀到的地面亮度：

    • 當中間感測器 (gs1) 偵測到低亮度（黑線），機器人會 轉向或停止

    • 若感測值為高亮度（白地），繼續直行

範例邏輯：

python
middle_value = gs1.getValue()
if middle_value < threshold: # 低亮度 = 黑線
turn_left()
else:
move_forward()


5. 為什麼要對 Ground Sensor 的數值進行校正（calibration）？

校正目的：

  • 每種地面材質、光照條件或貼圖會影響感測器回傳數值。

  • 相同的黑色/白色在不同模擬場景中可能對應不同數值。

  • 校正能確保感測值範圍穩定，例如：

    • 白地：約 1000（高反射）

    • 黑線：約 300（低反射）

方法：可透過顯示感測器數值（print()）實際量測並設定適當的閾值（threshold）。


6. 如何設定 Ground Sensor 的 sampling period？

可透過 enable(timestep) 設定每幾毫秒更新一次感測器數值：

python
timestep = int(robot.getBasicTimeStep())
ground_sensor.enable(timestep) # 每個模擬 step 更新一次數值
timestep = int(robot.getBasicTimeStep()) ground_sensor.enable(timestep) # 每個模擬 step 更新一次數值


7. e-puck 機器人可以偵測哪些顏色？如何判斷？

Ground Sensors 無法辨別真實色彩（RGB），只能量測「反射強度（亮度）」。

  • 可偵測「明暗差異」，例如：

    • 黑（低反射）：數值低

    • 白（高反射）：數值高

    • 灰：介於兩者之間

  • 若需判斷具體顏色（如紅、綠、藍），需使用 Camera + 色彩分析。


8. Tutorial3 如何讓 e-puck 在遇到黑線時做出反應？

主要邏輯如下：

  1. 透過中央感測器讀取地面亮度。

  2. 當亮度值低於設定門檻（即為黑線），改變輪子速度使機器人轉向或後退。

  3. 否則繼續前進。


9. 若地面顏色偵測判斷有誤，可能原因有哪些？

可能原因如下：

  • 閾值設定錯誤：未正確校正不同表面材質下的感測數值

  • 貼圖問題：地面材質或顏色不明顯，導致感測值變異不大

  • 感測器啟用錯誤：未設定 sampling period 或未啟用感測器

  • 程式邏輯錯誤：誤判或使用錯誤的感測器（如使用 gs2 卻以為是中央）

  • 模擬步長過長：感測器更新頻率太低，錯過偵測時機

Tutorial4 相關提問:


1. 如何讓機器人偵測牆壁或障礙物？

可使用 Webots 中的 距離感測器（DistanceSensor） 來達成。e-puck 機器人預設裝有 8 個紅外線距離感測器（名為 ps0 ~ ps7），可用來偵測牆壁、物體或障礙物的距離。

這些感測器通常分佈於機器人四周，涵蓋前、側、後方。


2. Webots 的距離感測器（DistanceSensor）如何運作？

  • 模擬的是紅外線反射感測器。

  • 根據物體與感測器之間的距離與材質反射特性，回傳一個模擬的數值。

  • 數值愈高，代表物體距離愈近；數值愈低，代表前方空曠。

注意：不同物體、角度、顏色會影響感測精度。


3. 如何在程式中啟用並讀取 DistanceSensor 的數值？

Python 範例：

python
sensors = []
for i in range(8):
sensor = robot.getDevice(f'ps{i}')
sensor.enable(timestep)
sensors.append(sensor)

while robot.step(timestep) != -1:
values = [s.getValue() for s in sensors]
print("Sensor readings:", values)


4. Tutorial4 中，e-puck 機器人如何實作避障行為？

基本策略：

  1. 偵測左右兩側感測器的數值（如 ps0~ps2 為右側，ps5~ps7 為左側）。

  2. 若右側感測值高（表示障礙靠右），機器人向左轉。

  3. 若左側感測值高，則向右轉。

  4. 若無障礙，則直行。

簡化邏輯（Python）如下：

python
left = max(ps5.getValue(), ps6.getValue(), ps7.getValue())
right = max(ps0.getValue(), ps1.getValue(), ps2.getValue())

if left > 80.0:
turn_right()
elif right > 80.0:
turn_left()
else:
move_forward()


5. 距離感測器數值的單位是什麼？如何轉換？

  • Webots 的 DistanceSensor.getValue() 回傳的是一個模擬的原始值，不是公尺或公分。

  • 數值範圍依感測器與模擬物件不同而異，一般 e-puck 紅外線感測器範圍約 0 到 1000 左右。

  • 若要轉為物理距離（例如公分），需要實測校正曲線或查閱原型感測器（如 e-puck 使用的 Sharp IR sensor）來建構換算關係。


6. 如何設定距離感測器的 sampling period？

與其他感測器相同，使用 enable(timestep) 來設定感測更新頻率。

python
timestep = int(robot.getBasicTimeStep())
ps0.enable(timestep)


7. 當機器人同時偵測到左、右兩側有障礙物時，應如何設計行為？

策略建議：

  • 停下來後退，再隨機選擇方向轉彎（避免陷入障礙中間）。

  • 使用中央感測器（如 ps3, ps4）判斷前方距離是否仍可通行。

  • 加入「優先方向」或「轉向權重」邏輯，例如偏向上次未轉的方向。

範例：

python
if left > 80 and right > 80:
move_backward()
if random() > 0.5:
turn_left()
else:
turn_right()


8. 為什麼避障時要考慮多個感測器的數值？

原因如下：

  • 障礙物形狀與方向多變，單一感測器可能無法偵測完整資訊。

  • 可避免誤判（如側面偵測不到但仍有障礙）。

  • 提高判斷精度，例如左右多個感測器數值加總、取最大值等。

  • 多感測器可提供方向性資訊（如障礙靠前左或後左）。


9. Tutorial4 若機器人無法正確避障，可能的解決方法有哪些？

可能原因與對應解法：

問題 解決方法
感測器未啟用 檢查程式是否 enable() 感測器
閾值設錯 實測感測器讀數並調整閾值
障礙物太遠 降低偵測觸發門檻（例如從 >100 改為 >80）
判斷邏輯錯誤 檢查左右邏輯是否顛倒
避障反應延遲 增加 sampling 頻率，減少步長
模型遮擋感測器 檢查感測器位置是否被模型本體或其他物件擋住
動作太慢 加快轉彎速度或改變轉彎策略

Tutorial5 相關提問:

如何讓機器人依照不同地面顏色或障礙物狀態做出不同反應？

策略：

結合地面感測器（Ground Sensor）與距離感測器（Distance Sensor）判斷環境，再做出對應反應。

範例邏輯：
python
ground = gs1.getValue()
left_obstacle = ps7.getValue()
right_obstacle = ps0.getValue()

if ground < 500: # 黑線
stop()
elif left_obstacle > 80 or right_obstacle > 80: # 有障礙物
avoid_obstacle()
else:
move_forward()

二、Tutorial 5 中如何結合多種感測器進行複雜任務？

Tutorial5 示範了如何利用：

  • 地面感測器（循線、辨識區域）

  • 距離感測器（避障）

  • 有限狀態機（控制任務流程）

應用：

機器人能同時巡線 → 遇障 → 閃避 → 回歸巡線 → 完成任務。

三、如何設計機器人巡線並同時避障？

這是典型的多任務整合問題，可採用優先權決策或狀態機（FSM）實作。

常見策略：

  1. 高優先權任務優先處理（如避障 > 巡線）

  2. 巡線：根據 gs1 為中心調整方向

  3. 若 ps0~ps2 或 ps5~ps7 偵測到障礙物則進入避障模式

四、若多種感測器回報衝突訊息，程式該如何決策？

建議使用 優先權架構或狀態機處理：

  • 避障為高優先 → 地面為中優先 → 其他行為為低優先

  • 也可使用 感測器融合機制加權平均 或 決策表

範例邏輯：

python
if obstacle_detected:
enter_avoid_mode()
elif line_detected:
follow_line()
else:
search_line()

五、在 Tutorial5 裡，機器人如何完成任務流程的狀態切換？

透過**有限狀態機（FSM）**來處理：

  • 定義狀態（如：IDLE, FOLLOW_LINE, AVOID, END）

  • 根據感測器輸入決定狀態轉移

範例結構：
python
state = "FOLLOW_LINE"

if state == "FOLLOW_LINE":
if obstacle_detected:
state = "AVOID"
else:
follow_line()
elif state == "AVOID":
if obstacle_cleared:
state = "FOLLOW_LINE"
else:
avoid_obstacle()

六、如何在程式中設計有限狀態機（FSM）？

設計步驟：

  1. 定義狀態集：列出所有行為狀態（如：IDLE、巡線、避障、完成）

  2. 建立轉移邏輯：設定感測條件與狀態間的轉移關係

  3. 用變數管理狀態：例如 state = "LINE_FOLLOW"

  4. 在主迴圈中處理行為與轉移

七、Tutorial5 的挑戰題目有哪些？如何著手解決？

常見挑戰：

  1. 在黑線中穿越障礙並返回路徑

  2. 抵達指定顏色區域後停下或發出訊號

  3. 通過不同顏色地面改變行為（例如紅色區轉彎、藍色區掉頭）

解法建議：

  • 使用地面感測器判斷顏色區域

  • 使用狀態機進行行為轉換

  • 加入路徑紀錄或尋徑策略協助回到原始路線

八、若要讓機器人自主完成一個完整路徑的巡邏任務，應加入哪些功能？

必要功能：

  • 路線循跡（巡線）

  • 障礙物偵測與避障

  • 區域辨識與轉向控制

  • 狀態管理（FSM）

  • 導航記憶（若需回到原點或重覆路徑）

  • （進階）路線地圖構建、SLAM、里程計

九、綜合 Tutorial3~5，請說明感測器資料融合（Sensor Fusion）的意義與應用

意義：

感測器資料融合（Sensor Fusion）指的是將來自不同感測器的輸入資訊整合，用以做出更穩定、準確、完整的判斷。

在 Tutorial3~5 中的應用：
感測器 功能 資料融合應用
Ground Sensor 判斷地面顏色/亮度 區域偵測、巡線
DistanceSensor 偵測障礙物距離 避障、判斷前方通行性
狀態機 管理多重任務流程 整合感測資訊 → 決策狀態切換
優點：

  • 減少單一感測器誤判

  • 提升環境理解與機器人自主性

  • 可實作複合任務（巡線 + 避障 + 任務完成）
